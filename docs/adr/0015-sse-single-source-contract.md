# ADR-0015: SSE 및 미확정 프로토콜 계약의 단일 정정 지점(Single Source) 설계

## Status
Accepted

## Context
Engram 및 외부 협력 모듈(예: Tauri Desktop, SSE transport) 간의 통신 사양은 개발 초기 단계에서 미확정 상태이거나 수시로 변경될 가능성이 높다. 이러한 프로토콜 계약(Schema / SSE payload 구조 등)을 코드베이스의 여러 위치에 하드코딩하거나 흩뿌려두면 다음과 같은 문제가 발생한다.
1. **중복 수정 및 누락**: 사양이 확정되거나 변경되었을 때, 다수의 파일(어댑터, 모커, 핸들러 등)을 일일이 찾아다니며 수정해야 하고, 이 과정에서 누락으로 인한 런타임 크래시가 발생할 수 있다.
2. **테스트 복잡성**: 각 모듈별로 서로 다른 Mock 데이터를 정의하게 되어, 정작 테스트 시에는 최신 사양을 반영하지 못하는 정합성 왜곡이 생긴다.
3. **토큰 소모 증가**: 에이전트가 코드를 탐색할 때 가변적인 계약 코드가 넓게 흩어져 있으면 컨텍스트 범위가 커져 불필요한 토큰 낭비가 발생한다.

## Decision
가변적인 외부 계약 및 SSE 프로토콜 Mock 사양은 **단일 파일(Single Source / Fixture / Constant 모듈)로 캡슐화하여 격리**한다.
1. **Mock Fixture 일원화**: SSE 페이로드, 이벤트 흐름, 기본 Mock 응답 셋은 프로젝트 루트 또는 전용 모듈(`src/api/fixtures.ts` 혹은 `src/api/contracts.rs`) 단 한 지점에서만 정의하고 관리한다.
2. **구현 무의존**: 실제 비즈니스 로직 및 전송 레이어는 구체적인 Mock 데이터 구조를 하드코딩하지 않고, 해당 단일 정정 지점에서 정의한 인터페이스/상수/Fixture 함수를 호출하여 작동하도록 설계한다.
3. **가이드라인 강제**: 후속 개발(예: #621 transport 연동 등)에서도 이 패턴을 동일하게 준수하여 실서버 SSE 라이브 적용 시의 코드 정정 범위를 최소화한다.

## Consequences
- **긍정**: 프로토콜 계약 사양이 라이브 서버 기준으로 확정되거나 변동될 때, 단 하나의 Fixture/계약 모듈 파일만 수정하면 되므로 재작업 범위와 기술 부채가 획기적으로 줄어든다.
- **긍정**: 전체 테스트 스위트에서 동일한 Mock 사양을 공유하여 정밀한 정합성 검증이 가능해진다.
- **부정**: 초기에 격리 레이어를 설계하고 Fixture 파일을 분리하는 데 소량의 선행 공수가 발생한다.
