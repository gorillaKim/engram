# 🔄 스프린트 및 회고(Retro) 운영

Engram은 애자일 스프린트 방법론을 도구 수준에서 탄탄하게 지원합니다. 
이 가이드에서는 스프린트를 계획하고 운영하는 과정과 스프린트 완료 후 자동 회고 리포트를 생성하는 방법에 대해 설명합니다.

---

## 📅 스프린트 생명주기 (Sprint Lifecycle)

스프린트는 보통 1~2주일 단위로 주기를 나누어 작업을 밀도 있게 집중하는 방법입니다.

1. **스프린트 생성 (Sprint Create)**
   * 새로운 스프린트를 이름과 목표 기간(시작일/종료일)을 설정하여 생성합니다.
   
2. **백로그 할당 (Planning)**
   * 진행 예정인 에픽과 이슈들을 활성화된 스프린트에 할당합니다.
   * `Ready` 상태의 이슈들이 스프린트 범위 내로 포함됩니다.

3. **스프린트 시작**
   * 스프린트가 시작되면 칸반보드 및 이슈 관리 뷰에서 현재 활성화된 스프린트 필터가 자동으로 적용되어, 팀이 집중해야 할 범위를 시각적으로 명확히 제한해 줍니다.

4. **스프린트 완료 (Sprint Complete)**
   * 스프린트 종료일에 도달하면 완료 처리를 실행합니다.
   * 이때 완료되지 않은 이슈(`Required`, `Ready`, `Working` 등)는 자동으로 다음 백로그로 이월되거나 다음 스프린트로 넘겨집니다.

---

## 📊 스프린트 회고 자동화 (`engram retro`)

스프린트가 끝나면 결과를 분석하고 회고하는 것이 중요합니다. Engram은 이를 위해 **`engram-retro`** 에이전트를 내장하고 있어, 쉘 명령어 한 줄로 완벽한 마크다운 기반 회고 리포트를 자동 추출해 줍니다.

### 회고 분석 항목
* **작업 완료율(Completion Rate)**: 계획된 대비 완료된 이슈 비율
* **스코프 팽창 감지(Scope Expansion)**: 스프린트 중간에 추가된 신규 이슈 분석
* **에이전트 기여도**: AI 에이전트가 완수한 이슈 및 투입된 시간 집계
* **병목 분석(Bottlenecks)**: 특정 이슈가 `Working` 상태에 오래 머무른 원인 및 의존성 관계 분석

### 사용법 (CLI)
터미널에서 아래의 명령어를 입력하면 현재 활성화된 스프린트 데이터를 분석하여 즉시 미려한 마크다운 리포트 파일로 추출해 줍니다.

```bash
# 현재 스프린트의 회고 문서 자동 생성
engram retro create

# 특정 스프린트 지정 분석 (예: 스프린트 ID 5)
engram retro create --sprint-id 5 --json
```

생성된 회고 문서는 프로젝트 내의 `.engram/retrospectives/` 디렉토리에 마크다운 형식으로 자동 보관되어, 깃으로 팀원들과 쉽게 공유하고 버전 관리할 수 있습니다.
